                # UX Writing

                A skill for writing product copy that guides users, reduces friction, and builds trust.

                ## Core principles

                **Clarity over cleverness.** Users are in task mode. Every word should help them move forward.

                **Human, not robotic.** Avoid corporate jargon and passive voice.
                Write like a knowledgeable friend, not a legal disclaimer.

                **Consistent voice.** Every screen is the same product speaking.
                Establish voice attributes early (e.g., "warm but efficient", "playful but trustworthy").

                **Contextual.** The best copy is invisible — it gives the user exactly what they
                need to know at the moment they need it, nothing more.

                ## Copy patterns by UI element

                ### Buttons and CTAs
                - Use verb + noun: "Save draft", "Send message", "Delete account"
                - Avoid context-free labels: "OK", "Yes", "Submit", "Click here"
                - For destructive actions, be explicit: "Delete forever" not "Delete"
                - Match the button label to the dialog question: "Discard changes?" then "Discard" / "Keep editing"

                ### Error messages
                - Say what happened in plain language
                - Say what the user can do about it
                - Never blame the user ("Invalid input" becomes "Please enter a valid email address")
                - Template: [What went wrong]. [How to fix it].
                - Bad: "Error 422: Validation failed"
                - Good: "That email address is already in use. Try signing in instead."

                ### Empty states
                - Explain why it's empty (first visit? cleared? filtered out?)
                - Tell the user what to do next (primary action)
                - Set expectations: what will appear here once it's populated?

                ### Onboarding and tooltips
                - One idea per tooltip
                - Lead with the benefit, not the feature name
                - Keep under 20 words when possible
                - Avoid "Click here to..." — they know how to click

                ### Form fields
                - Labels: nouns, not instructions ("Email" not "Enter your email address")
                - Placeholder text: only for format examples ("e.g. name@company.com")
                - Hint text: only when necessary — why we ask, expected format, where to find it
                - Inline validation: show while typing for email/password, not only on submit

                ### Confirmation and success messages
                - Confirm what happened, not just that something happened
                - Bad: "Done!"
                - Good: "Your changes have been saved."
                - For irreversible actions: "Your account has been permanently deleted."

                ### Modal dialogs
                - Title = what decision the user is making
                - Body = consequences and context (brief)
                - Buttons = the two choices, clearly labeled (not "Yes/No")

                ## Copy audit workflow

                When asked to review existing copy:
                1. Read all strings for tone and voice consistency
                2. Flag jargon, passive voice, vague language, or robotic phrasing
                3. Check all error messages follow the what/how-to-fix pattern
                4. Check all CTAs have verb + context (no "Submit", "OK", "Yes")
                5. Check form labels and hints for clarity
                6. Return a table: Current copy | Issue | Suggested revision
